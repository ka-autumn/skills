# ADR フォーマット

ADR は `docs/adr/` に配置し、連番を付けます: `0001-slug.md`、`0002-slug.md` など。

`docs/adr/` ディレクトリは遅延作成してください — 最初の ADR が必要になったときに初めて作ります。

## テンプレート

```md
# {決定の短いタイトル}

{1〜3 文で記述: どんな文脈で、何を決めたか、なぜそう決めたか。}
```

これだけです。ADR は 1 段落でも構いません。価値があるのは、決定が「行われたこと」と「なぜそうしたか」を記録することにあります — 各セクションを埋めることではありません。

## オプションのセクション

以下は本当に価値を加えるときだけ含めてください。ほとんどの ADR には不要です。

- **Status**(フロントマター)(`proposed | accepted | deprecated | superseded by ADR-NNNN`)— 決定が見直される際に有用
- **検討した選択肢** — 却下された代替案を覚えておく価値があるときのみ
- **影響** — 自明でない下流への影響を明示する必要があるときのみ

## 番号付け

`docs/adr/` をスキャンして既存の最大番号を見つけ、1 を加算します。

## ADR を提案すべきタイミング

以下の 3 つすべてが当てはまる必要があります:

1. **後戻りが難しい** — 後から考えを変える際のコストが意味のあるレベルで大きい
2. **文脈なしでは驚く** — 将来の読者がコードを見て「なぜこんなやり方をしたんだろう?」と疑問に思う
3. **本物のトレードオフの結果である** — 真の選択肢があり、特定の理由でそれを選んだ

簡単に元に戻せる決定であれば、記録は不要です — どうせ後で変えます。驚きがないなら、誰も疑問に思いません。本物の選択肢がなかったなら、「当然のことをした」以上に記録するものはありません。

### 該当するもの

- **アーキテクチャの形。**「モノレポを採用する。」「書き込みモデルはイベントソーシング、読み取りモデルは Postgres に投影する。」
- **コンテキスト間の統合パターン。**「Ordering と Billing は同期 HTTP ではなくドメインイベントで通信する。」
- **ロックインを伴う技術選定。** データベース、メッセージバス、認証プロバイダ、デプロイ先。すべてのライブラリではなく — 入れ替えに四半期かかるようなものだけ。
- **境界とスコープの決定。**「顧客データは Customer コンテキストが所有し、他のコンテキストは ID 経由でのみ参照する。」明示的な「やらない」は「やる」と同じくらい価値があります。
- **明らかな道筋からの意図的な逸脱。**「X の理由で ORM ではなく手書き SQL を使う。」 合理的な読者が逆を想定するようなもの。これらは次のエンジニアが意図的にそうしたものを「修正」するのを防ぎます。
- **コードから見えない制約。**「コンプライアンス要件のため AWS は使えない。」「パートナー API の契約のためレスポンスタイムは 200ms 以下でなければならない。」
- **却下された代替案のうち、却下の理由が自明でないもの。** GraphQL を検討し、微妙な理由で REST を選んだ場合は記録してください — そうしないと 6 ヶ月後にまた誰かが GraphQL を提案してきます。
