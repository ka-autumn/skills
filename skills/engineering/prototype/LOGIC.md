# ロジックプロトタイプ

ユーザーが手作業でステートモデルを動かせる、小さなインタラクティブなターミナルアプリ。問いが**ビジネスロジック、状態遷移、またはデータの形**に関するものであるとき —— 紙の上では妥当に見えるのに、実際のケースを通してみて初めて違和感が出る類のもの —— にこれを使用する。

## このアプローチが適しているケース

- 「このステートマシンが、X のあと Y というエッジケースをちゃんと処理できるか自信がない。」
- 「このデータモデルで、…というケースを実際に表現できるのだろうか。」
- 「書き始める前に、API がどうあるべきかを手で触って確かめたい。」
- ユーザーが**ボタンを押して状態が変わるのを見たい**と思うような場面すべて。

問いが「これはどう見えるべきか」であれば、アプローチが違う。[UI.md](UI.md) を参照してください。

## 手順

### 1. 問いを明文化する

コードを書く前に、どのステートモデルとどの問いをプロトタイプで検証しようとしているのかを書き留めてください。1 段落で、プロトタイプの README かファイル冒頭のコメントに記す。間違った問いに答えるロジックプロトタイプは、まったくの無駄である —— ユーザーが今その場で見ていても、AFK で後から戻ってくる場合でも、後から検証できるよう問いを明示すること。

### 2. 言語を選ぶ

ホストプロジェクトが使っているものを使う。プロジェクトに明確なランタイムがない場合（例：ドキュメントリポジトリ）は、確認すること。

ツールについてはプロジェクトの既存の規約に合わせること —— プロトタイプのためだけに新しいパッケージマネージャやランタイムを追加しないこと。

### 3. ロジックを可搬なモジュールに分離する

実際のロジック —— 問いに答えている部分 —— を、小さく純粋なインターフェースの背後に置く。後から抜き出して本物のコードベースに落とし込めるようにすること。その周りの TUI は使い捨てだが、ロジックモジュールはそうであってはならない。

適切な形は問いによって異なる：

- **純粋なリデューサー** —— `(state, action) => state`。アクションが離散的なイベントで、ステートが単一の値であるときに適している。
- **ステートマシン** —— 明示的な状態と遷移を持つ。「今この瞬間にどのアクションが正当か」が問いの一部であるときに適している。
- **少数の純粋関数** —— プレーンなデータ型に対して作用する。暗黙の現在状態がなく、変換だけがあるときに適している。
- **外部に見えるメソッドが明確なクラスまたはモジュール** —— ロジックが継続的な内部状態を本当に保持しているとき。

問いに最も合う形を選ぶこと。TUI に配線しやすいかどうかで選んでは*ならない*。純粋に保つこと：I/O なし、ターミナルコードなし、制御フローのための `console.log` もなし。TUI がそれをインポートして呼び出す。逆方向には何も流れない。

これこそが、プロトタイプを自身の寿命を超えて有用にする。問いに答えが出たら、検証済みのリデューサー／マシン／関数群を本物のモジュールへ抜き出せる —— TUI シェルは削除される。

### 4. ステートを可視化する最小限の TUI を作る

**軽量な TUI** として作ること —— ティックごとに画面をクリアし（`console.clear()` / `print("\033[2J\033[H")` / 同等のもの）、フレーム全体を再描画する。ユーザーには常に 1 つの安定した表示が見えるべきであり、際限なく伸びていくスクロールバックであってはならない。

各フレームは、次の順で 2 つの部分から成る：

1. **現在のステート**。差分を取りやすいよう整形して表示する（1 行に 1 フィールド、または整形済み JSON）。フィールド名やセクション見出しには **bold**、重要度の低いコンテキスト（タイムスタンプ、ID、導出値）には **dim** を使う。ネイティブの ANSI エスケープコードで構わない —— `\x1b[1m` で bold、`\x1b[2m` で dim、`\x1b[0m` でリセット。プロジェクトに既にスタイリングライブラリがある場合を除き、わざわざ導入する必要はない。
2. **キーボードショートカット**。最下部に一覧表示する：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。キーを bold に、説明を dim に（あるいはその逆に）—— 読みやすいほうにすること。

動作：

1. **ステートを初期化する** —— 単一のインメモリのオブジェクト／構造体。起動時に最初のフレームを描画する。
2. **一度に 1 つのキーストローク（または 1 行）を読み取り**、ステートを変更するハンドラへディスパッチする。
3. **再描画する** —— アクションのたびにフレーム全体を。追記ではなく置換すること。
4. **終了するまでループする。**

フレーム全体が 1 画面に収まるようにすること。

### 5. 1 コマンドで実行できるようにする

プロジェクトの既存のタスクランナー（`package.json` の scripts、`Makefile`、`justfile`、`pyproject.toml`）にスクリプトを追加すること。ユーザーは `pnpm run <prototype-name>` か同等のもので実行できるべきであり、パスを覚える必要が決してないようにする。

ホストプロジェクトにタスクランナーがない場合は、コマンドをプロトタイプの README の冒頭に記すだけでよい。

### 6. 引き渡す

ユーザーに実行コマンドを渡す。ユーザーは自分で操作する。興味深い瞬間は、ユーザーが「待って、これは起きてはいけないはずだ」や「あれ、X はもっと違うと思っていた」と言うときである —— それらは*アイデア*そのもののバグであり、それこそが狙いである。新しいアクションの追加を望まれたら、追加してください。プロトタイプは進化する。

### 7. 答えを記録する

プロトタイプがその役目を果たしたら、残す価値があるのは問いへの答えだけである。ユーザーがその場にいれば、何が分かったかを尋ねること。いなければ、プロトタイプの隣に `NOTES.md` を残し、プロトタイプが削除される前に答えを埋められるようにしておくこと（あるいは、あなたがセッションを見ていたなら、あなたが埋める）。

## アンチパターン

- **テストを追加しないこと。** テストを必要とするプロトタイプは、もはやプロトタイプではない。
- **本物のデータベースに配線しないこと。** 問いが永続化そのものに関するものでない限り、インメモリのストアを使うこと。
- **一般化しないこと。** 「あとで X もサポートしたくなったら？」は禁物。プロトタイプは 1 つの問いに答えるものである。
- **ロジックと TUI を混ぜて曖昧にしないこと。** リデューサー／ステートマシンが `console.log` やプロンプト、ターミナルのエスケープコードを参照していたら、それはもはや可搬ではない。TUI は純粋なモジュールの上の薄いシェルに留めること。
- **TUI シェルをプロダクションへ出荷しないこと。** シェルはターミナルから手で操作されることに最適化されている。残す価値があるのは、その背後にあるロジックモジュールである。
