# UI プロトタイプ

1 つのルート上に**構造的に大きく異なる複数の UI 案**を生成し、画面下部のフローティングバーで切り替えられるようにする。ユーザーはブラウザ上で案を切り替えながら、1 つを選ぶ（あるいは各案から良い部分を取り込む）か、残りを破棄する。

見た目ではなくロジック／ステートに関する問いであれば、アプローチが違う。[LOGIC.md](LOGIC.md) を参照してください。

## このアプローチが適しているケース

- 「このページはどう見えるべきか？」
- 「決める前に、このダッシュボードの案をいくつか見てみたい。」
- 「設定画面の別のレイアウトを試してみて。」
- そうでなければ、頭の中にある曖昧な 3 つのモックアップから選ぶのに丸 1 日費やしてしまう —— そんな場面すべて。

## 2 つのサブパターン —— サブパターン A を強く推奨する

UI プロトタイプは、**アプリの他の部分と隣り合わせになっている**ときのほうがはるかに判断しやすい —— 本物のヘッダー、本物のサイドバー、本物のデータ、本物の情報密度がある状態である。使い捨てのルートを単体で作ることは、コンテキストのない孤立した環境（真空状態）を作るようなものである。単体で見れば、どの案も問題ないように見えてしまうからである。案を組み込める妥当な既存ページが存在する場合は、常にサブパターン A をデフォルトの選択肢としてください。サブパターン B を選択するのは、プロトタイプを配置すべき適切な場所が周囲に本当に存在しない場合のみに留めてください。

### サブパターン A —— 既存ページの調整（推奨）

ルートはすでに存在している。各案は**同じルート上**に描画し、`?variant=` という URL の検索パラメータで出し分ける。既存のデータ取得・パラメータ・認証はすべてそのまま残り —— 描画部分だけが入れ替わる。これがデフォルトであり、そうしない特段の理由がない限りこれを選んでください。

まだページが存在しないものを対象とするプロトタイプでも、*それが自然に既存のページの内側に収まる*場合（ダッシュボードの新しいセクション、設定画面の新しいカード、既存フローの新しいステップなど）—— それも依然としてサブパターン A である。各案はホストとなるページの内側にマウントしてください。

### サブパターン B —— 新規ページ（最終手段）

これを使うのは、プロトタイプ対象が本当に収まる先となる既存ページを持たない場合に限ること —— 例えば、まったく新しいトップレベルの画面や、どこにも妥当に埋め込めないフローなどである。

プロジェクトが既に採用しているルーティング規約に従って、**使い捨てのルート**を作成してください —— 新しいトップレベル構造を考案しないこと。プロトタイプだと一目で分かる名前を付けること（例：パスやファイル名に `prototype` という語を含める）。`?variant=` のパターンは同じである。

サブパターン B に決める前に、念のため確認すること：これを埋め込める既存ページは本当に存在しないか？ 空のルートは、中身の詰まったルートなら露呈するはずの設計上の問題を覆い隠してしまう。

どちらのサブパターンでも、画面下部のフローティングバーは同一である。

## 手順

### 1. 問いを明文化し、N を決める

デフォルトは **3 案**とする。5 を超えると構造的に大きく異なるものではなくなり、ノイズになり始める —— そこを上限とすること。

計画を 1 行で、プロトタイプの場所かファイル冒頭のコメントに書き留めてください：

> 「`/settings` の既存ルート上で、`?variant=` で切り替え可能な、設定ページの 3 案。」

これは、ユーザーがその場にいて指摘できるかどうかにかかわらず有効である。

### 2. 構造的に大きく異なる案を生成する

各案を起草する。それぞれを次の基準に照らすこと：

- ページの目的と、そのページがアクセスできるデータ。
- プロジェクトのコンポーネントライブラリ／スタイリングシステム（TailwindCSS、shadcn、MUI、プレーンな CSS など何でも）。
- 明確にエクスポートされるコンポーネント名（例：`VariantA`、`VariantB`、`VariantC`）。

各案は**構造的に大きく異なっていなければならない** —— 異なるレイアウト、異なる情報階層、異なる主要なアフォーダンスであり、単に色が違うだけではいけない。少しずつ手を加えただけのカードグリッドが 3 つあっても、それは UI プロトタイプではなく、ただの壁紙である。2 つの草案が似すぎてしまった場合は、「カードグリッドを使わないこと」と明示的に指示して一方を作り直すこと。

### 3. これらをつなぎ合わせる

ルート上に、単一のスイッチャーコンポーネントを作成してください：

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

サブパターン A（既存ページ）の場合：既存のデータ取得はすべてスイッチャーより上に残し、案ごとに変わるのは描画されるサブツリーだけにすること。

サブパターン B（新規ページ）の場合：`/prototype/<name>` 下の使い捨てルートが、同じスイッチャーをマウントする。

### 4. フローティングスイッチャーを作成する

画面下部中央に配置する、固定位置の小さなバー。次の 3 つの要素から成る：

- **左矢印** —— 前の案へ切り替える（先頭まで来たら末尾へ回り込む）。
- **案ラベル** —— 現在の案のキーを表示し、案が名前をエクスポートしていればその名前も表示する。例：`B — Sidebar layout`。
- **右矢印** —— 次の案へ切り替える（末尾まで来たら先頭へ回り込む）。

動作：

- 矢印をクリックすると URL の検索パラメータを更新する（フレームワークのルーターを使う —— Next なら `router.replace`、React Router なら `navigate` など）。これにより、案を共有でき、リロードしても保持される。
- キーボード：`←`／`→` の矢印キーでも切り替えられる。ただし、`<input>`・`<textarea>`・`[contenteditable]` にフォーカスがあるときは矢印キーを横取りしないこと。
- ページから視覚的にはっきり区別できるようにすること（例：高コントラストのピル、控えめな影）。評価対象のデザインの一部でないことが一目で分かるようにする。
- プロダクションビルドでは非表示にすること —— `process.env.NODE_ENV !== 'production'` か同等のチェックで制御し、誤ってプロトタイプをマージしてもバーがユーザーに出荷されないようにする。

スイッチャーは単一の共有コンポーネントに入れ、両方のサブパターンで再利用できるようにすること。プロジェクトで共有 UI が置かれている場所に配置してください。

### 5. 引き渡す

URL（および `?variant=` のキー）を提示してください。ユーザーは手が空いたときに順に切り替えて見ていく。興味深いフィードバックは、たいてい **「B のヘッダーと C のサイドバーが欲しい」**といったものである —— それこそが、ユーザーが本当に求めているデザインである。

### 6. 答えを記録し、後片付けする

ある案が採用されたら、どれをなぜ選んだのかを書き留めること（コミットメッセージ、ADR、Issue、または AFK で動いていてユーザーがまだ応答していない場合はプロトタイプの隣の `NOTES.md`）。その後：

- **サブパターン A** —— 採用されなかった案とスイッチャーを削除し、採用案を既存ページに取り込む。
- **サブパターン B** —— 採用された案を本物のルートへ昇格させ、使い捨てルートとスイッチャーを削除する。

案のコンポーネントやスイッチャーを放置しないこと。これらはすぐに腐り、次に読む人を混乱させる。

## アンチパターン

- **色やコピーだけが違う案。** それは手直しであって、プロトタイプではない。本物の案は構造の点で食い違うものである。
- **案どうしでコードを共有しすぎる。** `<Header>` を共有するのは問題ないが、`<Layout>` を共有すると意味がなくなる。各案はレイアウトを捨てられる自由を持つべきである。
- **案を実際のミューテーションに配線する。** 読み取り専用のプロトタイプで構わない。案が変更を伴う必要がある場合は、スタブに向けること —— 問いは「これはどう見えるべきか」であって、「バックエンドが動くか」ではない。
- **プロトタイプを直接プロダクションへ昇格させる。** 案のコードはプロトタイプの制約（テストなし、最小限のエラーハンドリング）のもとで書かれている。取り込む際には、きちんと書き直すこと。
