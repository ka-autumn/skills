# PRD 化（to-prd）

このスキルは、現在の会話の文脈とコードベースの理解を PRD としてまとめる。ユーザーへのヒアリングは行わず、既に把握している内容を統合してください。

Issue トラッカーおよびトリアージラベルの語彙は事前に提供されているはずです。提供されていない場合は `/setup-matt-pocock-skills` を実行してください。

## 手順

1. まだコードベースの現状を把握していない場合は、リポジトリを探索してください。PRD 全体を通してプロジェクトのドメイン用語集の語彙を使い、対象領域に存在する ADR を尊重してください。

2. 機能（フィーチャー）をテストするためのシーム（結合点・テスト境界）をざっくり描き出してください。新しいシームよりも既存のシームを優先するべきです。可能な限り高い位置のシームを使用してください。新しいシームが必要な場合は、可能な限り高い位置で提案してください。

これらのシームがユーザーの想定と一致するか確認してください。

3. 後述のテンプレートに従って PRD を作成し、プロジェクトの Issue トラッカーに公開してください。トリアージラベルは `ready-for-agent` を付け、追加のトリアージは不要です。

## 課題（Problem Statement）

ユーザーが直面している課題を、ユーザーの視点で記述する。

## 解決策（Solution）

その課題に対する解決策を、ユーザーの視点で記述する。

## ユーザーストーリー（User Stories）

ユーザーストーリーを番号付きで長いリストとして列挙する。各ユーザーストーリーは次の形式に従うこと：

1. <アクター> として、<利益> のために、<機能> がほしい

<user-story-example>
1. モバイルバンクの顧客として、支出に関してより的確な判断ができるよう、自分の口座残高を確認したい
</user-story-example>

このユーザーストーリーのリストは極めて網羅的なものとし、機能のあらゆる側面をカバーすること。

## 実装上の決定事項（Implementation Decisions）

確定した実装上の決定事項を列挙する。次のような内容を含めてよい：

- 作成・修正されるモジュール
- 修正対象モジュールのインターフェース
- 開発者からの技術的な補足説明
- アーキテクチャ上の決定
- スキーマ変更
- API 契約
- 具体的なインタラクション

具体的なファイルパスやコードスニペットは**含めないこと**。すぐに陳腐化する可能性が高い。

例外：プロトタイプから得られたスニペット（ステートマシン、リデューサー、スキーマ、型の形状など）が散文よりも正確に決定内容を表現している場合は、対応する決定事項の中にそのままインラインで挿入し、プロトタイプ由来である旨を簡潔に添えること。動くデモではなく、決定の本質部分のみに切り詰めること。

## テストに関する決定事項（Testing Decisions）

確定したテストに関する決定事項を列挙する。次の内容を含める：

- 良いテストの条件についての記述（実装の詳細ではなく、外部から見える振る舞いのみをテストする）
- テスト対象となるモジュール
- テストの先行事例（コードベース内の類似テストなど）

## 対象外（Out of Scope）

この PRD において対象外とする事項を記述する。

## 補足（Further Notes）

機能に関するその他の補足事項。

</prd-template>
