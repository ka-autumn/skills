# トリアージ（Triage）

プロジェクトの Issue トラッカー上の Issue を、トリアージロールで構成されるシンプルなステートマシンに沿って分類・ステータスを設定する。

トリアージ中に Issue トラッカーへ投稿するコメントや Issue は、**必ず**以下の注意書きを冒頭に記載すること：

```
> *この内容はトリアージ中に AI によって生成されました。*
```

## 参考ドキュメント

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 変化に強いエージェント指示書の書き方
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` ナレッジベースの目的および運用ルール

## ロール

**分類**ロール 2 種：

- `bug` — 不具合が発生している
- `enhancement` — 新機能または改善

**ステータス**ロール 5 種：

- `needs-triage` — メンテナによる評価待ち
- `needs-info` — 報告者からの追加情報待ち
- `ready-for-agent` — 仕様が確定し、AFK エージェントが着手可能
- `ready-for-human` — 人間による実装が必要
- `wontfix` — 対応しない

トリアージ済みの Issue には、必ず**分類ロール 1 つとステータスロール 1 つ**を付けること。ステータスロールに矛盾が生じた場合は、何かを行う前にフラグを立ててメンテナに確認すること。

これらは正規ロール名であり、Issue トラッカー上で実際に使われるラベル文字列とは異なる場合がある。マッピングは事前に提供されているはず — されていなければ `/setup-matt-pocock-skills` を実行すること。

ステータス遷移：ラベル未付与の Issue は通常まず `needs-triage` に入り、そこから `needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix` のいずれかに遷移する。`needs-info` は報告者からの返信を受けて `needs-triage` に戻る。メンテナはいつでも上書き可能です — 不自然な遷移にはフラグを立て、進める前に確認すること。

## 起動方法

メンテナは `/triage` を起動し、やりたいことを自然言語で記述します。要求を解釈して行動してください。例：

- 「いま対応すべきものを教えて」
- 「#42 を確認したい」
- 「#42 を ready-for-agent に変更して」
- 「エージェントに任せられるのはどれ？」

## 対応が必要な Issue を一覧表示

Issue トラッカーを検索し、以下の 3 つのグループに分けてそれぞれ古い順で提示してください：

1. **未ラベル** — 一度もトリアージされていない。
2. **`needs-triage`** — 評価中。
3. **`needs-info` で、前回のトリアージノート以降に報告者による更新があったもの** — 再評価が必要。

件数と Issue ごとの 1 行サマリを表示し、メンテナに選んでもらってください。

## 対象の Issue をトリアージする

1. **コンテキストを収集。** Issue 全体（本文、コメント、ラベル、報告者、日時）を読んでください。過去のトリアージノートを解析し、解決済みの質問を蒸し返さないようにしてください。プロジェクトのドメイン用語集を使ってコードベースを探索し、関連領域の ADR を尊重してください。`.out-of-scope/*.md` を参照し、この Issue に類似する過去の却下事例を提示してください。

2. **提案。** 分類とステータスの提案を根拠と共にメンテナへ伝え、Issue に関連するコードベースの簡潔なサマリも添えてください。提案の回答を待ち、回答の指示に従ってください。

3. **再現確認（不具合の場合のみ）。** ヒアリングに入る前に、再現を試みてください：報告者の手順を読み、関連コードを追い、テストやコマンドを実行します。結果を報告してください — コードパス付きでの再現成功、再現失敗、または情報不足（強い `needs-info` シグナル）。再現確認はエージェント指示書を大きく強化できます。

4. **ヒアリング（必要な場合）。** Issue を肉付けする必要があれば、`/grill-with-docs` セッションを実行してください。

5. **結果を適用：**
   - `ready-for-agent` — エージェント指示書をコメントとして投稿してください（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — エージェント指示書と同じ構造ですが、なぜ委任できないかを明記してください（判断が必要、外部アクセス、設計判断、手動テスト等）。
   - `needs-info` — トリアージノートを投稿してください（後述のテンプレート）。
   - `wontfix`（不具合） — 丁寧な説明を投稿し、クローズしてください。
   - `wontfix`（機能要望） — `.out-of-scope/` に書き出し、コメントからリンクを貼ってからクローズしてください（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — ロールを適用してください。部分的な進捗があれば任意でコメントを記載してください。

## ステータスの即時上書き

メンテナが「#42 を ready-for-agent に変更して」と言った場合は、信頼してそのままロールを適用してください。何をしようとしているか（ロール変更、コメント、クローズ）を確認してから実行してください。ヒアリングはスキップします。ヒアリングなしで `ready-for-agent` に遷移させる場合は、エージェント指示書を書くかどうかメンテナに確認してください。

## Needs-info テンプレート

```markdown
## トリアージノート

**現時点で確定していること：**

- 項目 1
- 項目 2

**まだ確認が必要なこと（@{報告者のユーザー名}）：**

- 質問 1
- 質問 2
```

ヒアリング中に解決した内容はすべて「確定していること」の下に記録し、作業が失われないようにしてください。質問は具体的かつ実行可能なものに限ること。「詳細を教えてください」のような曖昧な質問は避けてください。

## 過去のセッションを再開する

Issue に過去のトリアージノートがあれば、それを読み、報告者が未解決の質問に回答しているかを確認し、続行する前に最新の状況を提示してください。解決済みの質問を蒸し返さないでください。
